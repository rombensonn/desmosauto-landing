# 06. Визуальные ассеты и 3D

## Роль визуалов

Визуалы в проекте не должны быть абстрактным украшением. Они помогают объяснять смысл:

- проблема клиента;
- решение через структуру сайта;
- доверие;
- тип услуги;
- SEO/AEO;
- путь к заявке.

Проект использует много bitmap/PNG 3D-моделей и реальных превью сайтов. Это один из признаков визуальной идентичности.

## Основные группы ассетов

### Problem models

Папки:

- `public/images/problem-models/`;
- `public/images/problem-models-uniform/`.

Смысл: визуализируют проблемы автосервисов и клиентов:

- повторные звонки;
- непонятная стоимость;
- недостаток доверия;
- отсутствие фото/гарантий;
- неудобная мобильная форма;
- сайт без роли в продажах.

### Solution models

Папки:

- `public/images/solution-models/`;
- `public/images/solution-models-uniform/`.

Смысл: показывают, как сайт решает проблемы:

- простое объяснение услуг;
- логика цены;
- короткая форма;
- доверие;
- подготовка к разговору;
- структурированные обращения.

### Service models

Папка:

- `public/images/service-models/`.

Смысл: отдельные форматы сайтов для разных автомобильных ниш.

### Trust models

Папка:

- `public/images/trust-models/`.

Смысл: аргументы доверия:

- авто-специализация;
- структура услуг;
- кейсы;
- быстрая демо-версия;
- PHP-форма;
- SEO/AEO;
- мобильный путь;
- передача сайта;
- контроль запуска.

### Project previews

Папка:

- `public/project-previews/`.

Это реальные превью рабочих проектов. Они особенно важны, потому что подтверждают обещание "120 рабочих кейсов".

## Правила использования ассетов

- Новые визуалы должны соответствовать существующей 3D-стилистике.
- Не заменять 3D-модели плоскими stock-иконками.
- Не использовать случайные декоративные SVG как главный визуал.
- У декоративных моделей `alt=""` и `aria-hidden="true"`.
- У смысловых скриншотов/превью должен быть понятный контекст.
- Использовать `assetPath()` для public assets, чтобы сборка работала с basePath.
- Для Next Image в статическом экспорте сохранять `unoptimized`.

## WebGL / R3F

Отдельная WebGL-сцена есть на AEO-направлении:

- файл блока: `app/aeo/AeoNeuroSearchBlock.tsx`;
- сцена: `app/aeo/AeoNeuroSearchScene.tsx`.

Сцена показывает нейро-выдачу и элементы:

- вопрос клиента;
- короткий ответ;
- доказательство;
- заявка.

Важно:

- всегда должен быть статический fallback;
- сцена не должна быть обязательной для понимания;
- reduced-motion должен отключать движение;
- canvas не должен быть интерактивным без отдельного запроса.

## Визуальное правило

Каждый визуал должен отвечать на вопрос: "Как это помогает владельцу автосервиса понять пользу сайта?" Если визуал не помогает, он не нужен.
