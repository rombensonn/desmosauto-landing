# 11. Правила качества и поддержки

## Главный принцип

Любое изменение должно сохранять три вещи:

1. продуктовый смысл;
2. визуальную систему;
3. техническую надежность.

Если изменение улучшает один слой, но ломает другой, его нужно пересобрать.

## Продуктовые правила

- Не превращать сайт в универсальный agency landing.
- Всегда сохранять авто-контекст.
- Не усложнять форму заявки.
- Не обещать гарантированный SEO-топ.
- Не удалять 120 кейсов как ключевое доказательство.
- Не скрывать главный CTA.
- Не ослаблять обещание "демо за сутки".

## Визуальные правила

- Сохранять нейтрально-светлую палитру + черный + оранжевый.
- Не вводить случайные новые цвета.
- Не добавлять декоративные blobs/orbs.
- Использовать существующие 3D-ассеты и preview-систему.
- Не ломать reduced-motion.
- Не делать hover-анимации слишком сильными.
- Не использовать карточки ради карточек.

## UX-правила

- Ошибки формы только после submit.
- Ошибка очищается при изменении поля.
- Согласия обязательны.
- Модалки должны закрываться по Escape и возвращать фокус.
- Форма должна сохранять человеческие сообщения об ошибках.
- Любой интерактивный элемент должен иметь состояние hover/focus/active/disabled/loading, если применимо.

## Технические правила

- Не редактировать `.next/`, `out/`, `output/` как исходники.
- Для production build всегда использовать `NEXT_PUBLIC_LEAD_ENDPOINT=/api/leads.php`.
- Не делать "готовый" архив без заполненного Telegram config.
- Не коммитить секреты.
- Проверять PHP синтаксис после изменения backend.
- Проверять build после изменений frontend.

## Правило обновления контекста

Если меняется:

- позиционирование;
- оффер;
- аудитория;
- дизайн-система;
- важная анимация;
- схема деплоя;
- backend формы;
- Telegram-логика;
- структура страниц,

то нужно обновлять эту учебную базу и/или `AGENTS.md`.

Контекст должен оставаться живым и точным. Устаревшие правила вреднее отсутствующих правил.

## Как будущая модель должна принимать решения

Перед изменением спросить себя:

1. Это помогает владельцу автосервиса получить заявки?
2. Это помогает клиенту понять услугу?
3. Это соответствует визуальному стилю?
4. Это не ломает форму и Telegram?
5. Это можно нормально собрать и загрузить на REG.RU?

Если ответ "нет" хотя бы на один ключевой вопрос, решение нужно пересмотреть.
