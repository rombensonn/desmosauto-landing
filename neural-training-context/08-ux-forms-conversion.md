# 08. UX, форма и конверсия

## Главный UX-фокус

Главный пользовательский путь - оставить заявку на демо. Весь интерфейс должен помогать человеку понять ценность и дойти до короткой формы.

## Форма заявки

Форма собирает:

- имя;
- телефон;
- согласие с политикой;
- согласие на обработку персональных данных;
- технический контекст страницы/источника/UTM;
- honeypot `website`.

Правила формы:

- не добавлять лишние поля без необходимости;
- не превращать заявку в длинную анкету;
- телефон форматируется под российский формат;
- ошибки появляются только после попытки отправки;
- при изменении поля его ошибка очищается;
- не валидировать на blur;
- сохранять введенные имя/телефон в sessionStorage;
- после успешной отправки очищать draft.

## Ошибки формы

Пользователь должен видеть понятное сообщение. Нельзя показывать:

- PHP stack trace;
- полный Telegram token;
- технические детали, которые не помогают пользователю.

Backend может логировать подробности в `storage/notification-errors.log`, но UI должен быть человекочитаемым.

## Согласия и юридические ссылки

Форма всегда должна содержать:

- ссылку на политику конфиденциальности;
- ссылку на согласие на обработку персональных данных;
- обязательные checkbox.

Это нельзя удалять, потому что форма собирает персональные данные.

## CTA

Главные CTA:

- "Получить демо за сутки";
- "Демо за сутки";
- "Смотреть 120 кейсов";
- "Оставить заявку".

CTA должны быть заметными, но не конкурировать друг с другом. Главный CTA - демо/заявка.

## Мобильный UX

Мобильная версия критически важна, потому что владелец или администратор может смотреть сайт с телефона.

Правила:

- кнопки полноширинные там, где это повышает удобство;
- текст не должен вылезать из карточек;
- фильтры проектов должны скроллиться горизонтально;
- длинные блоки должны становиться обычной сеткой/списком;
- форма должна быть простой и понятной.

## Доступность

Сохранять:

- semantic HTML;
- labels у полей;
- focus-visible;
- aria для модалок;
- role/aria у FAQ;
- reduced-motion;
- понятные error/success states.

## Конверсионное правило

Любой новый блок должен либо:

- объяснять ценность;
- закрывать возражение;
- показывать доказательство;
- помогать выбрать формат;
- вести к форме.

Если блок не делает ничего из этого, его не стоит добавлять.
